커밋 메시지는 주로 제목(Header), 본문(Body), 꼬리말(Footer) 3단 구조로 구성되며, 각 섹션은 빈 행으로 구분합니다. 핵심인 제목은 type: subject 형식으로 50자 이내의 명령문으로 작성하고, 상세 내용은 본문에 72자 이내로 기술하여 가독성을 높입니다.

[구조 예시]
text
type(scope): subject  # 필수, 헤더

body                  # 선택, 내용

footer                # 선택, 이슈 트래커 ID 등

1. 헤더 (Header) - 필수 
구조: 타입: 제목 형태 (예: feat: Add login api).
타입 (Type): 커밋의 성격 정의.
feat: 새로운 기능 추가
fix: 버그 수정
docs: 문서 수정
style: 코드 포맷팅 (코드 변경 없음)
refactor: 코드 리팩토링
test: 테스트 코드 추가/수정
chore: 빌드 업무 수정, 패키지 매니저 설정
제목 (Subject): 50자 이내로 핵심만, 영문인 경우 대문자로 시작하고 마침표(.)를 찍지 않음.

2. 본문 (Body) - 선택 
필수 요소는 아니지만, 무엇을, 왜 변경했는지 자세한 설명이 필요할 때 작성합니다.
각 행은 72자 이내로 제한하여 가독성을 유지합니다.

3. 꼬리말 (Footer) - 선택 
이슈 트래커 ID(예: Fixes: #123)나 브레이킹 체인지(Breaking Change)를 명시할 때 사용합니다.

4. 규칙 및 팁
제목과 본문은 한 줄 띄어서 구분합니다.
제목은 명령문으로 작성합니다 (예: Add (O), Added (X)).
Conventional Commits 컨벤션을 준수하면 자동화된 changelog 생성에 유리합니다.
